# 세션의 목적
주어진 코드를 리펙토링하려 한다. 아래의 지시에 따르라.

# 절차 (모든 단계를 반드시 순차적으로 따르라)
1.  **코드 구조 분석**
	- 내가 제공하는 코드를 아키텍처 관점에서 구조적으로 분석하라.
	- 모듈 구성, 레이어 구분, 의존 관계 등을 파악하라.
2.  **구체적인 문제점 진단**
	다음 항목별로 **구체적인 문제점**을 지적하라:
	- 설계 및 책임 분리 (SRP 위배 여부 등)
	- 추상화 수준과 중복도
	- 네이밍 일관성과 의도 전달력
	- 테스트 용이성 및 결합도
	- 유지보수성과 확장성
3.  **리팩토링 계획 수립 (코드 변경 전 단계)**
	- 단순 작업 목록이 아닌, **전반적인 리팩토링 방향성과 단계별 전략**을 제시하라.
	- 각 단계의 우선순위, 예상 리스크를 제시하라.
4.  **설계적 근거 제시**
	- 위 계획이 왜 타당한지 아래 원칙들을 바탕으로 설명하라:
		- SOLID, KISS, DRY
		- 클린 아키텍처 / 계층형 구조 / 도메인 중심 설계
		- 테스트 전략 및 의존성 분리
5.  **대안적 방향 제시 (선택 사항)**
	- 가능하다면 다른 설계 접근 방식도 제안하라.
	- 각각의 장점/단점을 간결하게 요약하라.
6.  **사용자의 확인 후 다음 단계 진행**
	- 사용자의 동의 없이 다음 단계로 넘어가지 말 것.
	- 진행하기 전 반드시 "진행해도 될까요" 라고 물어볼 것.
7.	**유니티 에디터에서 추가로 필요한 작업 예상**
	- 변경 이후, 이전과 같이 작동하기 위해 유니티 에디터에서 추가 작업이 예상된다면 알려라.
	- 가능하면 추가 작업이 필요 없는 방향으로 재설계할 것.

# 리팩토링 철학 (모든 판단의 기준으로 삼을 것)
- 비즈니스와 도메인에 맞는 책임 분리
- 테스트 가능한 구조와 의존성 분리
- 의도가 명확히 드러나는 코드
- 미래의 요구 변경에 유연하게 대응할 수 있는 설계
